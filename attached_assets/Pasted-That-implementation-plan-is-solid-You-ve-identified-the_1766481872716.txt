The "Mate Distance" Correction (Phase 3)This is the #1 cause of engines playing "suicide" moves or missing mate.The Problem: If the bot finds a "Mate in 5" at Depth 10, it stores that score. Later, it sees the same position at Depth 8. If you just return the stored score, the bot thinks it’s still a Mate in 5 from the current position, which is mathematically impossible.The Fix: Store "Absolute Mate Scores."Storing: If the score is a mate, add the current ply to it before saving: scoreToStore = score + ply.Retrieving: When you hit a TT match, subtract the ply: scoreToUse = storedScore - ply.This ensures "Mate in 5" always means "Mate in 5 moves from the current board," regardless of when it was first found.2. The 3 Bound Flags (The "Traffic Lights")In your Minimax/Negamax loop, you must handle the alpha/beta window carefully when saving:FlagConditionMeaningEXACTalpha < score < betaWe found the precise value of this position.LOWERBOUNDscore >= betaThis move is "too good" (Beta Cutoff). The real score is at least this high.UPPERBOUNDscore <= alphaThis move is "too bad" (All-Node). The real score is at most this low.3. Depth-Preferred Replacement (Phase 2)Since memory isn't infinite, your "Replacement Scheme" determines how "smart" the memory is.Strategy: If a new result has a higher depth than what is currently in the TT slot, always replace it. Deep searches are exponentially more valuable than shallow ones.Aging (Pro-tip): Add an age or gameID counter. If you start a new game, the bot shouldn't be using "memories" from the previous game that might have had a different piece-square table or personality.4.
