1. The "Listening Loop" Conflict
When you add multiple informational commands, you risk the bot and the user "talking over" each other.

The Guard: Ensure every informational command (like "What is the time?") explicitly calls your new stopAndWait() method before the bot speaks.

The Resume: After the bot finishes telling the user the time or material, you must trigger resumeAfterTTS() so the microphone "wakes up" to hear the next move without the user having to touch the phone.

2. Ambiguity in "Find Items"
When implementing "Where is my [Piece]?", remember that players often have two of the same piece.

The Logic: If a user asks "Where is my Rook?", and they have two, the bot should not just pick one. It should say: "You have Rooks on a1 and f1".

The Filter: Ensure your search logic only looks for the player's color, not the bot's pieces, to avoid accidentally giving away the opponent's position (which would be actual cheating).

3. The "Eval" and "Peek" Penalty Box
To defend your app's integrity as a trainer, these two commands need to be "flagged" in your gameStats.ts.

Statistical Impact: If a user uses "Evaluation" or "Show Board," you should set a flag like wasAssisted = true for that game.

Clarity Score: You might want to automatically cap the Clarity Score at 50% for any game where the user asked the engine for an evaluation, as they didn't rely solely on their mental board.

4. Natural Language for "Material Balance"
As we discussed, avoid a robotic list.

Implementation: Use a helper function to group pieces (e.g., "2 Pawns" instead of "Pawn, Pawn").

The "Net" approach: Users usually only care about the difference. If both sides have a Queen, don't mention the Queen. Only mention what is "extra" (e.g., "White is up a Bishop").

5. "Legal Moves" Performance
The "Legal moves for [Piece]" command can be long-winded if a piece has 10+ options.

The UX Fix: If a piece has more than 4 legal moves, have the bot say: "The Queen has 12 legal moves, including d4, e5, and f6." This gives the user the "anchor" they need without overwhelming them with a 30-second list of coordinates.

6. The "Show Board" Safety Timer
For the voice-triggered "Show board" command, do not rely on the user to say "Hide."

The Guard: Hard-code a setTimeout of 5 seconds in your game.tsx that automatically sets setIsPeeking(false). This ensures the "Blindfold" nature of the app is maintained even if the user forgets to speak the "Cover" command.

7. Haptic Confirmation (S10+)
Since the user is playing eyes-free, they need a non-vocal confirmation that their command was heard.

The Pulse: Use Haptics.impact({ style: ImpactStyle.Light }) the moment a command like "Time" or "Material" is recognized. This tells the user "I'm processing your request" before the bot even starts talking.
